# Yivi Verzekeringen — OpenID4VP/4VCI insurance demo

A demo showing how an insurer can:

1. **Identify** a customer using their digital ID / passport (OpenID4VP, x509 certificate trust).
2. **Issue** a pseudonymous customer card (OpenID4VCI, DID trust).
3. **Re-authenticate** the customer on a return visit using only that pseudonym (OpenID4VP, DID trust) — no name or date of birth disclosed.

The fictional insurer is **Yivi Verzekeringen**. The UI is in Dutch.

## Quick start

The entire stack runs locally in Docker. No staging dependency for the demo flow itself.

```sh
./scripts/gen-verifier-cert.sh    # one-time, generates compose/eudi-verifier/keystore.p12
docker compose up
./scripts/adb-reverse.sh          # forwards local ports to a USB-tethered Android wallet
open http://localhost:5173
```

### One-time setup before the first demo

The local EUDI verifier is configured with `X5CShouldBe: Ignored` so it accepts any passport credential — including passports issued by the **staging** PID Issuer (`eudiw-pid-issuer.openid4vc.staging.yivi.app`). Use the staging PID Issuer's web flow to install a passport (Tyler Neal, DOB 1955-04-12) into your wallet once, then the local demo works.

The local EUDI verifier identifies itself with a self-signed x509 cert (SAN `localhost`) generated by `scripts/gen-verifier-cert.sh`. The wallet has to be told to trust this cert — load `compose/eudi-verifier/localhost.crt` into the wallet's verifier trust anchors (see the Yivi EUDI wallet docs for how).

## Architecture (local)

```
host machine
├── docker compose
│   ├── postgres            (Veramo DB, schemas: agent + verifier)
│   ├── veramo-issuer       :18081 → issues InsurancePseudonymCredential
│   ├── veramo-verifier     :18082 → verifies InsurancePseudonymCredential
│   ├── eudi-verifier       :18083 → verifies passport / ID card
│   ├── backend             :3001  → /api/registreren, /api/inloggen
│   └── frontend            :5173  → Vite dev server (HMR enabled)
└── adb reverse (Android phone) → host :18081-:18083, :5173, :3001
```

- Service-to-service traffic stays on the docker network (`http://veramo-issuer:3000`, etc.).
- Wallet-facing URLs are `http://localhost:<port>` so the phone wallet reaches them via `adb reverse` on the loopback interface.
- Veramo Issuer + Verifier both use `did:jwk` locally (no `did:web` HTTP resolution to debug).
- Frontend (Vite HMR) and backend (`tsx watch`) hot-reload from `./frontend/src` and `./backend/src` mounts.

## Demo flow

### `/registreren`

1. **Identificatie** — wallet discloses `firstName`, `lastName`, `dateOfBirth` over OpenID4VP (cert trust, EUDI verifier).
2. **Polisnummer** — user enters their polisnummer. The backend strictly verifies that the polisnummer's stored name + DOB match the disclosed identity. A "Polisnummer invullen voor dit ID-bewijs" button autofills the matching polisnummer for demo smoothness.
3. **Klantenpas** — backend generates a random `YIVI-INS-XXXXXXXX` pseudonym, links it to the customer record in `db.json`, and issues an `InsurancePseudonymCredentialSdJwt` over OpenID4VCI (DID trust, Veramo Issuer). Wallet stores the card.

### `/inloggen`

1. Wallet discloses the `insurance_id` from the `InsurancePseudonymCredential` over OpenID4VP (DID trust, Veramo Verifier).
2. Backend looks up the customer record by `insurance_id` and shows "Welkom terug, Tyler Neal · polisnummer 12345678 · klant sinds 2018".

### Seeded customers

`backend/db.seed.json` ships three customers:

| Polisnummer | Naam              | DOB        | Klant sinds |
| ----------- | ----------------- | ---------- | ----------- |
| 12345678    | Tyler Neal        | 1955-04-12 | 2018-03-14  |
| 87654321    | Erika Mustermann  | 1964-08-12 | 2020-09-01  |
| 11223344    | Jan Janssen       | 1972-11-30 | 2021-06-15  |

Tyler matches the staging PID Issuer's pre-seeded keycloak user (`openid4vc-poc-ops/environments/dev/keycloak/pid-issuer-users-0.json`), so out of the box the demo works with a wallet that's been loaded from staging.

## Development

- Edit `backend/src/*.ts` → `tsx watch` picks it up, the backend container restarts automatically.
- Edit `frontend/src/*.tsx` → Vite HMR updates the browser without reload.
- Edit `compose/veramo-issuer/conf/*` → restart the issuer (`docker compose restart veramo-issuer`).
- Reset the customer DB to seed: `docker compose down -v && docker compose up`.
